# Visual and Doc Standards

## 1. 視覺標準：先讀懂，再精緻
優先順序永遠是：
1. 主代理目前狀態
2. 主代理所在房間
3. 最近事件是否正常
4. subagents / sessions 是否異常增長
5. 最後才是裝飾與氛圍

## 2. 借鏡 Star Office 的正確方式
可以學：
- world-first 首屏
- 狀態到區域映射
- 房間與家具提升語意
- 角色沿動線移動

不要學成：
- 直接複製場景構圖
- 直接照搬文案與產品人格
- 為了像 reference 而破壞 Hermes 專案語意

## 3. 文件分層標準
- `PROJECT_MAP.md` 只做入口與索引
- `architecture/` 描述系統關係，不塞實作細節
- `modules/` 描述責任、邊界、風險
- `conventions/` 記錄標準與規範
- `references/` 放外部參考與對照
- 單檔盡量控制在 200 行內，超出就拆檔

## 4. 寫作準則
- 高層寫「為什麼」與「邊界」
- 低層寫「有哪些檔案」與「誰負責什麼」
- 多用連結，少重複貼相同敘述
- 若規則變成實作共識，更新 spec，不只更新 README

## 5. 視覺演進準則
- 新視覺若不能提升判讀，就不應加入
- 新面板若和世界視圖重複，應先刪面板內容
- 手機與窄畫面要優先保留主代理與主要狀態
- 新資產優先選擇授權清楚的開放素材
